# 技能管理系统改造方案

## 方案对比

### 方案 A: 完全迁移到 Claude Skills (推荐)

**做法**: 将所有数据库技能转换为文件系统技能，删除数据库技能管理。

**优点**:
- ✅ 符合 Claude Skills 开放标准
- ✅ 文件系统技能更易维护 (版本控制、协作编辑)
- ✅ 渐进式披露机制完整
- ✅ 跨平台可移植
- ✅ 简化架构，单一技能系统

**缺点**:
- ❌ 需要迁移现有8个技能
- ❌ 前端需要改造 (如果依赖数据库管理界面)

**实施步骤**:
1. 为8个数据库技能创建 SKILL.md 文件
2. 创建参考文档和脚本
3. 更新前端以读取文件系统
4. 逐步下线数据库技能管理

---

### 方案 B: 混合模式 - 数据库管理元数据，文件系统存储内容

**做法**: 保留数据库存储技能元数据，但实际内容从文件系统读取。

**架构**:
```
Database (Metadata)          File System (Content)
┌─────────────────┐         ┌─────────────────┐
│ skills 表        │─────────>│ skills/ 目录     │
│ - name           │  指向    │ - SKILL.md       │
│ - description   │         │ - reference/     │
│ - file_path      │         │ - scripts/       │
│ - intent_keywords│         │                 │
└─────────────────┘         └─────────────────┘
```

**优点**:
- ✅ 保留数据库的管理能力 (查询、过滤)
- ✅ 符合 Claude Skills 的内容存储
- ✅ 前端改动较小

**缺点**:
- ❌ 仍然需要维护两套系统
- ❌ 同步复杂度增加
- ❌ 不完全符合 Claude Skills 标准

---

### 方案 C: 保持现状 + 同步工具

**做法**: 维持两套系统，但提供同步工具保持一致性。

**优点**:
- ✅ 无需大规模改造
- ✅ 渐进式演进

**缺点**:
- ❌ 维护成本持续存在
- ❌ 架构复杂度不降低

---

## 推荐方案：方案 A (完全迁移)

### 迁移设计

```
阶段 1: 创建文件技能 (1-2天)
├── 创建 skills/ 目录结构
├── 为8个数据库技能创建 SKILL.md
├── 创建 reference/ 和 scripts/ 子目录
└── 验证功能完整性

阶段 2: 数据库技能只读 (1天)
├── 禁用数据库技能的编辑功能

阶段 3: 前端改造 (2-3天)
├── 添加文件系统技能编辑界面
├── 支持在线编辑 SKILL.md

阶段 4: 数据迁移 (1天)
├── 将用户配置数据迁移到新格式
├── 下线数据库技能表
└── 清理旧代码

阶段 5: 优化 (1-2天)
├── 添加技能验证工具
├── 实现技能版本控制
└── 性能优化
```
